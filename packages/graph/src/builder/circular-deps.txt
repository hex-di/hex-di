IMPORT PATTERN
==============

4 parallel type modules need GraphBuilder:
  - inspection.ts
  - provide.ts
  - merge.ts
  - init-order-types.ts

All 4 import from builder.ts (as "../builder.js")

builder.ts imports types/index.ts which re-exports from all 4 modules.

Result: 4 separate circular import chains converging at types/index.ts

STATUS: 2 of 4 Cycles Fixed via GraphBuilderSignature
═════════════════════════════════════════════════════

FIXED (using GraphBuilderSignature pattern):
  - inspection.ts - Uses GraphBuilderSignature for pattern matching only
  - init-order-types.ts - Uses GraphBuilderSignature for pattern matching only

CANNOT BE EASILY FIXED:
  - provide.ts - Needs GraphBuilder as RETURN TYPE (not pattern matching)
  - merge.ts - Needs GraphBuilder as RETURN TYPE (not pattern matching)

Why these 2 remain unfixed:
  - ProvideResult and MergeResult return GraphBuilder<...> as the type
  - GraphBuilderSignature lacks methods like .provide(), .build(), etc.
  - Using GraphBuilderSignature as return type would break method chaining
  - The provide() method's return needs to support further .provide() calls

TypeScript Tolerance:
These remaining imports are TYPE-ONLY (using "import type" syntax), which
TypeScript handles specially:

1. Type-only imports are erased at compile time
2. No actual code execution creates cycles
3. TypeScript's declaration merging allows these patterns
4. The cycles do NOT cause runtime errors

HOWEVER, Consider:
  - IDE performance may suffer with complex type cycles
  - Type-checking speed could slow with deep cycles
  - Some bundlers or other tools may struggle

Potential Future Fix:
One approach would be to make GraphBuilderSignature include method signatures
that return `this` type, but this would create additional complexity and
potentially new circular dependencies.
