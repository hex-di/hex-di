# 6.12 Glossary — GxP Compliance

> **Part of:** [GxP Compliance (§6)](./README.md) | **Previous:** [§6.11 FMEA Risk Analysis](./11-fmea-risk-analysis.md) | **Next:** [§7 Integration](../07-integration.md)

> For the shared glossary of technical and regulatory terms, see [../../cross-cutting/gxp/13-glossary.md](../../cross-cutting/gxp/13-glossary.md). This section defines clock-specific terms not covered in the cross-cutting glossary.

## Purpose

This glossary defines the technical and regulatory terms used throughout the `@hex-di/clock` specification. It is intended for non-technical regulatory reviewers, auditors, and quality assurance personnel who may not be familiar with the software-specific terminology.

---

## Technical Terms

| Term                            | Definition                                                                                                                                                                                                                                                                                                                                                               |
| ------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Adapter**                     | A concrete implementation of a port interface. For example, `SystemClockAdapter` is an adapter that implements `ClockPort` using platform timing APIs. Multiple adapters may exist for the same port (e.g., `SystemClockAdapter` for production, `VirtualClockAdapter` for testing, `NtpClockAdapter` for GxP).                                                          |
| **Canonical input**             | The deterministic JSON string that was hashed to produce a cryptographic digest. Persisted alongside the digest for auditor inspectability — allows verification of exactly what was hashed without re-implementing the canonicalization logic. See `TemporalContextDigest.canonicalInput`.                                                                                |
| **Clamped fallback**            | A timing mechanism that wraps `Date.now()` to prevent backward time jumps. When `Date.now()` returns a value less than the previous call (due to NTP correction), the clamped fallback returns the previous value instead. This provides monotonicity without `performance.now()`.                                                                                       |
| **Clock drift**                 | The observed divergence between the system wall-clock time and an NTP reference time server. Monitored by ecosystem GxP infrastructure to detect when timestamps may be inaccurate beyond acceptable thresholds.                                                                                                                                                         |
| **Coarsened**                   | Reduced timer resolution applied by the browser or runtime for security reasons (e.g., Spectre mitigations). A coarsened timer rounds timestamps to a lower precision (e.g., from microseconds to milliseconds), meaning two events occurring close together may receive identical timestamps.                                                                           |
| **Composite key**               | A unique identifier formed by combining two or more fields. In `@hex-di/clock`, the composite key `(processInstanceId, sequenceNumber)` uniquely identifies an event across all process lifetimes and restarts.                                                                                                                                                          |
| **Constant-time comparison**    | A comparison algorithm that takes equal time regardless of whether inputs match, preventing timing side-channel attacks. Used by `verifyTemporalContextDigest()` to compare digest values without leaking information about which bytes differ.                                                                                                                           |
| **DI container**                | Dependency Injection container. A runtime system that manages the creation and distribution of objects (services, adapters) in an application. `@hex-di/clock` adapters are registered in the DI container and injected into consumers that depend on `ClockPort`.                                                                                                       |
| **Epoch**                       | A reference point in time from which timestamps are measured. In JavaScript, the epoch is January 1, 1970, 00:00:00 UTC (Unix epoch). `wallClockNow()` returns the number of milliseconds since this epoch.                                                                                                                                                              |
| **FIPS (Federal Information Processing Standards)** | A set of standards developed by the US National Institute of Standards and Technology (NIST) for use in computer systems by non-military government agencies and government contractors. FIPS 180-4 defines SHA-256. FIPS 140-2/140-3 defines security requirements for cryptographic modules. GxP organizations with FIPS requirements MUST verify their deployment runtime's `crypto` module is configured in FIPS mode. |
| **Frozen object**               | A JavaScript object on which `Object.freeze()` has been called. Properties of a frozen object cannot be added, removed, or modified at runtime. This provides runtime immutability (not cryptographic tamper-evidence).                                                                                                                                                  |
| **High-resolution time**        | Time measurement with sub-millisecond precision, typically using `performance.timeOrigin + performance.now()`. Provides epoch-referenced timestamps with microsecond or better precision on supported platforms.                                                                                                                                                         |
| **HMAC (Hash-based Message Authentication Code)** | A specific type of message authentication code involving a cryptographic hash function and a secret key. Not used directly by `@hex-di/clock` (which uses plain SHA-256 digests for tamper-evidence), but referenced in the context of ecosystem monitoring adapters that may use HMAC for authenticated hash chains. |
| **Injectable**                  | A design pattern where dependencies are provided to a component from the outside rather than being hard-coded. An injectable clock means the timing source can be changed without modifying the consuming code.                                                                                                                                                          |
| **ISO 8601**                    | International standard for date and time representation. `@hex-di/clock` externalizes all timestamps in ISO 8601 UTC format (e.g., `2026-02-14T12:00:00.000Z`) for human-readable audit reports and cross-system interoperability. The `Z` suffix denotes UTC timezone.                                                                                                  |
| **Microtask**                   | A JavaScript execution queue for asynchronous operations (e.g., `Promise.then()` callbacks). Microtasks interleave with synchronous code within the same event loop tick. OQ-3 tests verify that `next()` produces unique sequence numbers even when called from concurrent microtasks.                                                                                   |
| **Monotonic time**              | A time source that only moves forward and is never adjusted backward. `performance.now()` provides monotonic time that is immune to NTP corrections or manual clock adjustments. Monotonic time is relative (elapsed time since an arbitrary origin), not absolute (calendar time).                                                                                      |
| **Mutation testing**            | A software testing technique that introduces small changes (mutations) to source code and verifies that existing tests detect and fail on each mutation. Measured as a mutation score (percentage of mutations killed). `@hex-di/clock` targets >95% mutation score using Stryker Mutator.                                                                                 |
| **NTP (Network Time Protocol)** | A protocol for synchronizing computer clocks over a network. NTP daemons (e.g., `chrony`, `ntpd`) continuously adjust the system clock to match reference time servers, ensuring wall-clock accuracy within milliseconds.                                                                                                                                                |
| **NTP step correction**         | A large, instantaneous clock adjustment applied by the NTP daemon when the system clock has drifted significantly. Step corrections can cause `Date.now()` to jump forward or backward, which is why monotonic time is preferred for duration measurement.                                                                                                               |
| **Leap smear**                  | A technique for handling leap seconds by distributing the one-second adjustment over a longer period (e.g., 24 hours) rather than inserting or deleting a single second. Prevents sudden one-second jumps in `Date.now()`. Configured in the NTP daemon.                                                                                                                 |
| **Platform API capture**        | The technique of storing references to `Date.now`, `performance.now()`, and `performance.timeOrigin` in closure scope at adapter construction time. Prevents post-construction tampering — even if an attacker replaces `globalThis.Date.now`, the adapter continues using the originally captured reference.                                                              |
| **Port**                        | An interface that defines a contract for a capability without specifying how it is implemented. In hexagonal architecture, ports are the boundary between the application's core logic and its external dependencies. `ClockPort` is a port; `SystemClockAdapter` is an adapter that implements it.                                                                      |
| **Process instance identifier** | A unique identifier for multi-process deployments (e.g., horizontally scaled Node.js workers). Combined with sequence numbers as the composite key `(processInstanceId, sequenceNumber)` to provide globally unique event ordering across process lifetimes and restarts. See §3.3.                                                                                      |
| **Result type**                 | A discriminated union type (`Result<T, E>`) from `@hex-di/result` that represents either success (`ok(value)`) or failure (`err(error)`). All fallible operations in `@hex-di/clock` return `Result` instead of throwing exceptions, making error paths explicit at the type level and eliminating try/catch in consumer code.                                            |
| **Safe integer**                | A JavaScript integer within the range `Number.MIN_SAFE_INTEGER` to `Number.MAX_SAFE_INTEGER` (±2^53 - 1) where arithmetic operations are exact. Sequence numbers are constrained to safe integers; overflow is detected when `next()` would exceed `Number.MAX_SAFE_INTEGER`.                                                                                            |
| **Schema version**              | A version field included in serialized `TemporalContext` records enabling forward-compatible deserialization across library upgrades. Deserialization utilities validate the schema version and reject unknown versions, ensuring archived records remain interpretable (ALCOA+ Enduring).                                                                                 |
| **Sentinel value**              | A special value used to signal an exceptional condition. In `@hex-di/clock`, the sequence number `-1` is a sentinel indicating a sequence overflow event. Sentinel values are chosen to be outside the normal value range (sequence numbers normally start at 1).                                                                                                        |
| **SHA-256**                     | A cryptographic hash function producing a 256-bit (32-byte) digest. Used by `computeTemporalContextDigest()` to generate per-record tamper-evidence hashes of `TemporalContext` records. The implementation uses platform-native `crypto.createHash('sha256')`, not JavaScript polyfills.                                                                                 |
| **Startup self-test**           | A series of automated validation checks (ST-1 through ST-5) performed synchronously during `createSystemClock()` construction. Verifies that platform timing APIs return plausible values and that GxP platform freeze requirements are met. Satisfies 21 CFR 11.10(h) device check requirements.                                                                        |
| **Structural irresettability**  | A design pattern where the inability to perform an operation (e.g., resetting a sequence counter) is enforced at the type level (compile time), not just at runtime. The production `SequenceGeneratorPort` interface does not include a `reset()` method, making it impossible for any code to call `reset()` on a production sequence generator, regardless of intent. |
| **TemporalContext**             | A frozen data structure containing three fields — `sequenceNumber`, `monotonicTimestamp`, and `wallClockTimestamp` — produced by `TemporalContextFactory.create()`. Serves as the standardized audit timestamp type for all HexDI ecosystem packages, ensuring consistent temporal data capture across the system.                                                        |
| **Time skew**                   | The difference between `highResNow()` and `wallClockNow()` caused by `performance.timeOrigin` drift. Detected by startup self-test ST-5 when the divergence exceeds 1000ms, indicating the platform's high-resolution clock is unreliable.                                                                                                                               |
| **Timing side-channel**         | An attack technique that extracts information by measuring the time taken by a computation. For example, a naive byte-by-byte comparison of two hash digests returns faster when the first byte differs, leaking information about which bytes match. `verifyTemporalContextDigest()` uses constant-time comparison to prevent timing side-channel attacks on digest verification. |
| **Wall-clock time**             | Calendar time as displayed on a wall clock, representing the current date and time. Obtained from `Date.now()` in JavaScript. Subject to NTP corrections and manual adjustments, unlike monotonic time.                                                                                                                                                                  |
| **Auto-advance**                | A VirtualClockAdapter feature where each time read (`monotonicNow()`, `wallClockNow()`, `highResNow()`) automatically advances the internal clock state by a configurable number of milliseconds. Returns the value before advancement. Controlled by `setAutoAdvance()` / `getAutoAdvance()`.                                                                            |
| **BlockUntil**                  | A synchronization primitive on VirtualTimerScheduler that returns a promise resolving when the pending timer count reaches or exceeds a specified threshold. Uses real (platform) time for its timeout to prevent deadlocks. Ensures async operations have registered their timers before test code advances virtual time.                                                   |
| **Branded type (phantom)**      | A TypeScript pattern where a plain type (e.g., `number`) is intersected with a unique symbol brand (e.g., `number & { readonly [MonotonicBrand]: true }`) to create a nominally distinct type at compile time with zero runtime cost. Used to prevent cross-domain timestamp misuse.                                                                                     |
| **CachedClockPort**             | A structurally separate port (not extending `ClockPort`) that provides periodically-refreshed time values for ultra-high-throughput scenarios. Method names differ from `ClockPort` (`recentMonotonicNow` vs. `monotonicNow`) to prevent accidental substitution. MUST NOT be used for audit trail timestamps.                                                             |
| **ClockTimeoutError**           | An error type thrown when `VirtualTimerScheduler.blockUntil()` times out waiting for the expected number of pending timers. Contains `expected` (target count), `actual` (count at timeout), `timeoutMs` (timeout duration).                                                                                                                                              |
| **Covariant widening**          | A type-theoretic property where a subtype can be used wherever its supertype is expected. Branded timestamps (e.g., `MonotonicTimestamp`) are subtypes of `number` and can be assigned to `number` variables, but not to other branded types (e.g., `WallClockTimestamp`).                                                                                                 |
| **TimerHandle**                 | An opaque, frozen object returned by `TimerSchedulerPort.setTimeout()` and `setInterval()`. Contains a `_tag: "TimerHandle"` discriminant and a numeric `id`. Used with `clearTimeout()` / `clearInterval()` for cancellation. Consumers SHOULD NOT extract the `id` for platform API use.                                                                               |
| **TimerSchedulerPort**          | An injectable port for scheduling future work (`setTimeout`, `setInterval`, `clearTimeout`, `clearInterval`, `sleep`). Separate from `ClockPort` per Single Responsibility Principle. Enables deterministic timer testing via `VirtualTimerScheduler`.                                                                                                                    |
| **VirtualTimerScheduler**       | A test-only `TimerSchedulerPort` adapter linked to a `VirtualClockAdapter`. When virtual time advances, pending timers fire synchronously in chronological order (FIFO for ties). Provides `pendingCount()`, `advanceTime()`, `runAll()`, `runNext()`, and `blockUntil()` for deterministic timer testing.                                                                |
| **Retention metadata**          | A frozen utility type (`RetentionMetadata`) providing standardized fields for data retention policy composition with audit trail records. Fields: `retentionPeriodDays`, `retentionBasis`, `retentionStartDate`, `retentionExpiryDate`, `recordType`. Validated by `validateRetentionMetadata()`.                                                                          |
| **Incident classification**     | A severity-based categorization (L1–L4) of clock infrastructure incidents. L1: operational anomaly, no confirmed impact. L2: potential validated state impact. L3: confirmed validated state compromise. L4: immediate risk to patient safety or system-wide failure. See § 6.7 (recovery-procedures.md) for full definitions.                                               |
| **Compensating control**        | An alternative procedure or mechanism that provides equivalent risk mitigation when the primary control is unavailable. In `@hex-di/clock`, CLK-GXP-008 defines compensating controls for FM-3 through FM-6 detection when the ecosystem monitoring adapter is not co-deployed.                                                                                            |
| **Recovery verification**       | A set of test specifications (RV-1 through RV-7) that validate the effectiveness of recovery procedures by simulating each failure mode and verifying the corresponding recovery restores normal operation. Executed during OQ or PQ qualification.                                                                                                                        |
| **Disaster recovery**           | Extended test scenarios (DR-1 through DR-4) that validate end-to-end recovery capability under realistic failure conditions (process crash under load, NTP server failure, storage failure during write, platform API corruption). Executed annually by GxP organizations.                                                                                                   |

## Regulatory Terms

| Term                                         | Definition                                                                                                                                                                                                                                                                            | Reference                                         |
| -------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------- |
| **21 CFR Part 11**                           | US FDA regulation governing electronic records and electronic signatures in pharmaceutical and medical device industries. Defines requirements for audit trails, access controls, and system validation.                                                                              | FDA, Code of Federal Regulations Title 21 Part 11 |
| **ALCOA+**                                   | Data integrity framework: Attributable, Legible, Contemporaneous, Original, Accurate (ALCOA) plus Complete, Consistent, Enduring, Available. Used to assess whether electronic records meet data integrity requirements.                                                              | WHO, PIC/S, FDA guidance documents                |
| **CAPA (Corrective and Preventive Action)**  | A systematic process for investigating the root cause of a non-conformance, implementing corrective actions to address the immediate issue, and implementing preventive actions to prevent recurrence. Required for clock-related failures under emergency change control procedures.  | 21 CFR 820.90, ICH Q10                           |
| **CSV (Computerized System Validation)**     | The documented process of establishing and maintaining evidence that a computerized system meets its intended purpose and complies with applicable regulations. Includes IQ, OQ, PQ protocols.                                                                                        | EU GMP Annex 11, GAMP 5                           |
| **Data retention period**                    | The minimum time an organization must keep clock-derived audit records. Varies by record type and regulatory context: 21 CFR 211.180 (1 year post-expiry for pharmaceuticals), 21 CFR 820.180 (2 years post-release for devices), ICH E6(R2) (2 years post-approval for clinical).   | 21 CFR 211.180, 21 CFR 820.180, ICH E6(R2)       |
| **DQ (Deployment Qualification)**            | Verification that the infrastructure surrounding a computerized system (network, NTP, hardware) meets the requirements for correct operation. Complements IQ/OQ/PQ by covering environment-level concerns.                                                                            | Local convention per GAMP 5                       |
| **Electronic record**                        | Any combination of text, graphics, data, audio, pictorial, or other information in digital form that is created, modified, maintained, archived, retrieved, or distributed by a computer system. Timestamps are a component of electronic records.                                    | 21 CFR 11.3(b)(6)                                 |
| **Electronic signature**                     | A computer data compilation of any symbol or series of symbols executed, adopted, or authorized by an individual to be the legally binding equivalent of the individual's handwritten signature.                                                                                      | 21 CFR 11.3(b)(7)                                 |
| **Electronic signature binding**             | The cryptographic association of a signer's identity with a specific temporal context and record. In `@hex-di/clock`, implemented via `SignableTemporalContext` and enforced by `validateSignableTemporalContext()` per 21 CFR 11.50 and 11.70.                                       | 21 CFR 11.50, 21 CFR 11.70                        |
| **EU GMP Annex 11**                          | European Union Good Manufacturing Practice guideline for computerized systems used in pharmaceutical manufacturing. Covers validation, data integrity, access control, change management, and incident management.                                                                    | European Commission, EudraLex Volume 4 Annex 11   |
| **FMEA (Failure Mode and Effects Analysis)** | A systematic method for identifying potential failure modes in a system, evaluating their effects, and prioritizing them by Risk Priority Number (RPN = Severity x Occurrence x Detection).                                                                                           | ICH Q9, GAMP 5                                    |
| **GAMP 5**                                   | Good Automated Manufacturing Practice, version 5. A risk-based framework for computerized system validation in the pharmaceutical industry. Categorizes software into 5 categories (1-5) based on configurability and customization. `@hex-di/clock` is Category 5 (custom software). | ISPE GAMP 5 Guide                                 |
| **GxP**                                      | A general abbreviation for "Good x Practice" regulations and guidelines (GMP, GLP, GCP, GDP, etc.) governing pharmaceutical, biotechnology, and medical device industries. The "x" represents the specific practice area. GxP compliance requires documented evidence that systems meet regulatory requirements.                                                         | FDA, EMA, WHO                                     |
| **ICH Q7**                                   | International Council for Harmonisation guideline on Good Manufacturing Practice for Active Pharmaceutical Ingredients. Section 6.5 covers equipment calibration, applicable to timing equipment.                                                                                     | ICH Q7                                            |
| **ICH Q9**                                   | International Council for Harmonisation guideline on Quality Risk Management. Defines principles and tools (including FMEA) for assessing and managing quality risks.                                                                                                                 | ICH Q9                                            |
| **IQ (Installation Qualification)**          | Documented verification that a system is installed correctly and according to its specification. For `@hex-di/clock`, IQ verifies package installation, export surface, immutability, platform detection, and startup self-test.                                                      | GAMP 5, EU GMP Annex 11                           |
| **OQ (Operational Qualification)**           | Documented verification that a system operates correctly under representative conditions. For `@hex-di/clock`, OQ verifies monotonicity under load, accuracy under load, concurrency uniqueness, and precision.                                                                       | GAMP 5, EU GMP Annex 11                           |
| **PQ (Performance Qualification)**           | Documented verification that a system performs acceptably under sustained real-world conditions on specific deployment hardware. For `@hex-di/clock`, PQ verifies throughput, sustained precision, extended uniqueness, and memory stability.                                         | GAMP 5, EU GMP Annex 11                           |
| **RFC 2119**                                 | Internet Engineering Task Force standard defining the interpretation of requirement-level keywords MUST, MUST NOT, SHALL, SHALL NOT, SHOULD, SHOULD NOT, MAY, and OPTIONAL in specification documents. Used throughout this specification to express mandatory vs. advisory requirements. | IETF RFC 2119                                     |
| **RPN (Risk Priority Number)**               | A numerical score (1-1000) calculated as Severity x Occurrence x Detection in an FMEA. Higher RPNs indicate higher-priority risks requiring mitigation.                                                                                                                               | ICH Q9                                            |
| **RTM (Requirements Traceability Matrix)**   | A document that maps each regulatory requirement to the specification section(s) that address it, implementation artifacts that realize it, and validation tests that verify it. Ensures forward and backward traceability between requirements and evidence. See §6.8.               | EU GMP Annex 11 Section 4, GAMP 5 Appendix M4    |
| **SOP (Standard Operating Procedure)**       | A documented procedure that describes how to perform a specific task consistently. GxP environments require SOPs for failure recovery, change control, and system operation.                                                                                                          | 21 CFR 211, EU GMP                                |
| **Validation plan**                          | An organization's documented strategy for establishing that a computerized system meets its intended use. For `@hex-di/clock`, includes IQ/OQ/PQ execution scope, parameter justification, PQ threshold rationale, and regulatory evidence collection procedures.                    | EU GMP Annex 11 Section 4, GAMP 5                 |
