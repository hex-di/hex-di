# Glossary

Domain terminology used throughout the SpecForge specification.

## Core Concepts

**ACP (Agent Communication Protocol)**
An open standard for agent-to-agent communication. SpecForge adopts ACP as its primary agent protocol. SpecForge is dual-mode: an ACP server (exposing agent roles to external clients via `GET /agents`, `POST /runs`) and an ACP client (invoking agent runs for flow orchestration). See [ADR-018](./decisions/ADR-018-acp-agent-protocol.md).

**ACP Agent Manifest**
A declarative description of an agent's capabilities, served via `GET /agents`. Each SpecForge agent role is registered as an ACP agent manifest containing name, description, input/output content types, and skill declarations. See [BEH-SF-210](./behaviors/BEH-SF-209-acp-server.md).

**ACP Await**
A mechanism where an agent yields execution to pause and wait for external input (human feedback, clarification, approval). The run enters `awaiting` state until resumed via `POST /runs/{id}/resume`. Replaces the previous approval gate and clarification patterns. See [BEH-SF-221](./behaviors/BEH-SF-219-acp-client.md), [BEH-SF-232](./behaviors/BEH-SF-229-acp-messaging.md).

**ACP Run**
A single execution of an agent within the ACP protocol. Follows a 7-state lifecycle: `created` → `in_progress` → `completed`/`failed`/`awaiting`/`cancelling`/`cancelled`. Each run belongs to an ACP session and produces ACP messages. See [BEH-SF-212](./behaviors/BEH-SF-209-acp-server.md).

**ACP Session**
A persistent context container for ACP runs. Holds append-only message history and state descriptors (URL-based, supporting distributed storage). Each flow run gets its own ACP session, enabling context accumulation across iterations. See [BEH-SF-224](./behaviors/BEH-SF-219-acp-client.md), [BEH-SF-236](./behaviors/BEH-SF-229-acp-messaging.md).

**AgentRole**
A tagged identifier for a specialized agent function within a flow (e.g., `spec-author`, `reviewer`, `dev-agent`). Each role has a system prompt, tool set, model selection, and permission tier. Roles are declared in flow definitions and resolved by the Agent Registry. See [types/flow.md](./types/flow.md), [behaviors/BEH-SF-017-agent-roles.md](./behaviors/BEH-SF-017-agent-roles.md).

**AgentHandle** _(Superseded)_
See `ACPRunHandle`. Replaced by [ADR-018](./decisions/ADR-018-acp-agent-protocol.md) (ACP Agent Protocol).

**Agent Backend**
An execution engine behind the ACP protocol layer that translates ACP runs into actual LLM invocations.

**Agent Plan**
A structured progress-tracking primitive sent as a `FlowUpdate { _tag: "AgentPlan" }` variant. Contains an array of `PlanEntry` items, each with `content`, `priority` (high/medium/low), and `status` (pending/in_progress/completed). Plans are complete replacements — the agent sends all entries in each update; the client replaces the current plan entirely. Enables progress tracking UI without parsing free-form text. See [ADR-019](./decisions/ADR-019-zed-inspired-architecture.md), [types/acp.md](./types/acp.md#agent-plan).

**Agent Registry**
Unified resolution service for agent role manifests across all sources: built-in roles (loaded at startup), dynamic templates (generated by activation predicates), and marketplace agents (fetched from cloud). All sources produce `ACPAgentManifest` entries resolved by role ID. Replaces the former `AgentHandlerRegistry` component. See [ADR-019](./decisions/ADR-019-zed-inspired-architecture.md), [types/ports.md](./types/ports.md#agent-registry-port). The default implementation is `ClaudeCodeBackend`, which wraps `claude -p`. Other backends can be plugged in via `AgentBackendPort`. See [behaviors/BEH-SF-239-agent-backend.md](./behaviors/BEH-SF-239-agent-backend.md).

**Agent Session**
A long-lived agent execution context with a defined system prompt, role-scoped tools, and accumulated conversation context. Unlike stateless function calls, agent sessions persist throughout a flow, building understanding over multiple iterations. Managed via ACP sessions and runs.

**Approval Gate**
Per-phase configuration requiring human sign-off before the flow proceeds to the next phase. When a `PhaseDefinition` has `requiresApproval: true`, the phase pauses after convergence and waits for explicit human approval via `specforge approve` before the next phase begins. See [BEH-SF-125](./behaviors/BEH-SF-121-human-in-the-loop.md).

**Blackboard** _(Superseded — see ACP Session, Message Exchange)_
Formerly a session-scoped shared workspace with three layers (documents, findings, messages). Superseded by ACP messaging: documents are now `ACPMessagePart` artifacts, findings use `CitationMetadata`, and messages are role-addressed `ACPMessage` instances. See [ADR-018](./decisions/ADR-018-acp-agent-protocol.md).

**Blast Radius Analysis**
Computation of maximum impact scope when an agent requests elevated permissions. Includes transitive closure of affected files, behaviors, and tests via graph relationships, producing an impact score. See [BEH-SF-208](./behaviors/BEH-SF-201-permission-governance.md).

**CodeDiff**
A first-class content type in the `FlowUpdate` union representing a structured code change. Contains `path`, `oldText`, and `newText` fields. Translates to `GraphNode { type: 'FileDiff' }` in the knowledge graph with `MODIFIED_BY` → session and `AFFECTS` → File node edges. Enables rich diff rendering in the dashboard without parsing free-form text. See [ADR-019](./decisions/ADR-019-zed-inspired-architecture.md), [types/acp.md](./types/acp.md#flowupdate).

**Collective Memory**
Aggregated knowledge patterns across all agent roles and sessions for a project. Synthesized after every flow completion by merging, ranking, and resolving conflicting patterns. Feeds into CLAUDE.md generation. See [BEH-SF-184](./behaviors/BEH-SF-177-memory-generation.md).

**Compliance Gate**
A PreToolUse hook that validates tool invocations against structural and formatting rules before execution. Exit code 2 blocks non-compliant writes. In GxP mode, compliance gates additionally block destructive git operations and mandate review gates. See [BEH-SF-166](./behaviors/BEH-SF-161-hook-pipeline.md).

**Composed Context**
A new session context bootstrapped from selected session chunks across prior sessions.

**Connection**
A live subprocess link to an agent backend, managed by `ConnectionManagerService`. Separated from sessions per [ADR-019](./decisions/ADR-019-zed-inspired-architecture.md) — a session holds conversation state; a connection holds the subprocess handle. Connections can be pooled and reused across sessions (via `keepAlive`). Has three states: `initializing`, `ready`, `closed`. See [types/acp.md](./types/acp.md#connections), [types/ports.md](./types/ports.md#connection-manager-port). Created by querying the knowledge graph for relevant chunks (by topic, agent role, or similarity) and assembling them into a coherent starting context for a new agent session. This is the mechanism for knowledge transfer across flow runs.

**Compositional Session**
The architectural pattern where agent session conversations are materialized in Neo4j as chunks, enabling them to be queried, selected, and composed into new session contexts. This creates knowledge continuity — prior reasoning is reusable, not lost when a session ends.

**Convergence**
The condition under which a flow phase terminates. Defined by convergence criteria: metrics thresholds (e.g., zero critical findings, 80% coverage) evaluated after each iteration. A phase runs until converged or until it hits its maximum iteration bound.

**Distributed Context Window**
The conceptual model where the knowledge graph, ACP session history, and active agent sessions collectively form a distributed context. No single agent holds all knowledge; instead, the graph provides persistent memory, ACP sessions provide scoped working memory, and agents provide specialized reasoning.

**Dynamic Role Factory**
A registry of `RoleTemplate` definitions with activation predicates (Cypher queries) evaluated against the project graph at flow start. Matching templates generate specialized agent roles automatically based on detected languages, frameworks, and dependencies. See [BEH-SF-185](./behaviors/BEH-SF-185-dynamic-agents.md).

**Export Adapter**
Pluggable adapter for rendering knowledge graph data to files (markdown, ADR, coverage report). Implements the `ExportAdapterPort` interface with `render()` and `supports()` methods. New export formats can be added by implementing the adapter and registering with the export registry. See [BEH-SF-131](./behaviors/BEH-SF-127-import-export.md).

**Finding**
A review issue yielded as an ACP message with `CitationMetadata` by a reviewer agent. Each finding has a severity (`critical`, `major`, `minor`, `observation`) and a status (`open`, `resolved`, `wont-fix`, `deferred`). Findings reference requirement IDs and spec files for traceability. See [BEH-SF-230](./behaviors/BEH-SF-229-acp-messaging.md).

**Flow**
A declarative sequence of phases that orchestrates agent sessions to accomplish a goal. Flows define which agent roles participate in each phase, convergence criteria, and execution order. Flows are the primary unit of work orchestration in SpecForge. SpecForge ships predefined flows and supports custom flow definitions.

**Flow Definition**
A declarative data structure (not imperative code) that specifies a flow's phases, agent roles per phase, convergence criteria, and dependencies. Registered with the flow engine and discoverable by name.

**FlowUpdate**
A closed discriminated union of 17 tagged variants representing session-scoped streaming updates. Replaces the generic `ACPMessage` callback in `MessageExchangeService.subscribe()`. Consumers exhaustively match on `_tag`. Original 12 variants: `AgentMessageChunk`, `AgentThoughtChunk`, `AgentFinding`, `AgentDocument`, `ToolCallStarted`, `ToolCallCompleted`, `CodeDiff`, `AgentPlan`, `PermissionRequested`, `InterAgentMessage`, `BudgetZoneChanged`, `PhaseConverged`. Plus 5 new variants from ADR-020/022: `ToolCallProgress`, `ExtFlowUpdate`, `CommandsUpdate`, `ConfigOptionsUpdate`, `StopReasonUpdate`. Distinct from `OrchestratorEvent` which covers system-level lifecycle events. See [ADR-019](./decisions/ADR-019-zed-inspired-architecture.md), [ADR-020](./decisions/ADR-020-protocol-extension-observability.md), [ADR-022](./decisions/ADR-022-dynamic-agent-capabilities.md), [types/acp.md](./types/acp.md#flowupdate).

**Flow Input**
The parameters for starting a flow run: target package, spec path, source path, diff reference, scope, composition context, and execution options (batch mode, max iterations, GxP, model override, token budget).

**Flow Preset**
Predefined configuration (quick/standard/thorough) controlling max iterations, model selection, and convergence strictness. Presets allow users to trade cost and time for thoroughness without manually configuring each flow parameter.

**Flow Run**
A single execution of a flow definition. Has its own ACP session, session manager scope, and graph sync context. Identified by a unique flow run ID (`flowRunId`). See [BEH-SF-236](./behaviors/BEH-SF-229-acp-messaging.md).

**GxP Mode**
An optional compliance mode that activates audit trail hash chains, agent invocation records, user identity tracking, data retention policies, document governance, FMEA risk analysis, traceability enforcement, and validation protocols (IQ/OQ/PQ). Disabled by default.

**Human Feedback**
Human-injected ACP messages with `role: "user"` and `priority: "highest"` metadata, processed by the `feedback-synthesizer` above agent findings. Posted via `specforge feedback` CLI command or the web dashboard's feedback input box. Delivered by resuming awaiting ACP runs. See [BEH-SF-121](./behaviors/BEH-SF-121-human-in-the-loop.md), [BEH-SF-233](./behaviors/BEH-SF-229-acp-messaging.md).

**Import Adapter**
Pluggable adapter for importing external specifications (markdown, OpenAPI) into the knowledge graph. Implements the `ImportAdapterPort` interface with `parse()` and `supports()` methods. New import formats can be added by implementing the adapter and registering with the import registry. See [BEH-SF-130](./behaviors/BEH-SF-127-import-export.md).

**Neo4j**
An open-source graph database used by SpecForge as the canonical data store. Stores specification artifacts, code topology, session history, and their relationships as nodes and edges. Queried using the Cypher query language. Connected via the Bolt protocol. Mode-switched: local instance (solo) vs AuraDB managed cluster (SaaS).

**Knowledge Graph**
The Neo4j database that stores all specification artifacts (requirements, invariants, ADRs, tasks, findings), code topology (source files, symbols, packages), session history (agent sessions, session chunks), and their relationships. The canonical data store — filesystem documents are renderings of graph data.

**Knowledge Transfer Pipeline**
Process of extracting `KnowledgePattern` items from materialized session chunks, classifying them by type, and making high-confidence patterns available for CLAUDE.md generation. See [BEH-SF-180](./behaviors/BEH-SF-177-memory-generation.md).

**Model Escalation**
Automatic upgrade from a cheaper model (sonnet) to a more capable model (opus) after N iterations when findings remain unresolved. Configured via `modelStrategy: "escalating"` and a `ModelEscalation` configuration on a phase definition. See [BEH-SF-080](./behaviors/BEH-SF-073-token-budgeting.md).

**Plugin**
Self-contained extension providing agent roles, conventions, hooks, and/or flow definitions. Discovered from `~/.specforge/plugins/` (global) and `.specforge/plugins/` (project-local) directories. Each plugin declares a `PluginManifest` describing its capabilities and activation mode. See [BEH-SF-090](./behaviors/BEH-SF-087-extensibility.md).

**Role Template**
A parameterized agent definition in the dynamic role factory. Contains activation predicate, system prompt template with `{{paramName}}` slots, tool set, and model selection. Instantiated when its Cypher predicate evaluates to true against the project graph. See [BEH-SF-186](./behaviors/BEH-SF-185-dynamic-agents.md).

**Structured Output**
JSON schema-validated agent output replacing text parsing. Each agent role produces typed graph nodes directly via Claude Code's `--json-schema` flag, including `graphNodes[]`, `graphEdges[]`, `findings[]`, `errors[]`, and `selfAssessment`. See [ADR-012](./decisions/ADR-012-json-first-structured-output.md).

## Agent Roles

**Discovery Agent** (`discovery-agent`)
Conducts Phase 0 discovery: interactive conversation with the user to gather requirements, research context, and produce a requirements brief. See [BEH-SF-017](./behaviors/BEH-SF-017-agent-roles.md).

**Spec Author** (`spec-author`)
Consolidated authoring agent that handles the full spec document lifecycle: scaffolding initial structure from the requirements brief, writing detailed spec content (behaviors, constraints, port definitions, error types), revising based on reviewer feedback, finalizing for handoff, and generating specs from reverse-engineered codebase data. See [BEH-SF-018](./behaviors/BEH-SF-017-agent-roles.md).

**Reviewer** (`reviewer`)
Consolidated review agent that handles architectural review (layering, coupling, API design) and traceability checking (requirement-to-implementation chain validation). Produces findings with severity ratings. See [BEH-SF-019](./behaviors/BEH-SF-017-agent-roles.md).

**Feedback Synthesizer** (`feedback-synthesizer`)
Aggregates findings from reviewers and human feedback into prioritized action items for authors. Human feedback is ranked above all agent findings. See [BEH-SF-020](./behaviors/BEH-SF-017-agent-roles.md).

**Task Decomposer** (`task-decomposer`)
Decomposes spec requirements into ordered, dependency-aware implementation task groups with acceptance criteria. See [BEH-SF-021](./behaviors/BEH-SF-017-agent-roles.md).

**Dev Agent** (`dev-agent`)
Consolidated implementation agent that handles code implementation, test execution, and implementation verification in a single persistent session. Reads task groups, implements code, runs tests, and iteratively repairs failures. See [BEH-SF-022](./behaviors/BEH-SF-017-agent-roles.md).

**Codebase Analyzer** (`codebase-analyzer`)
Consolidated reverse engineering agent that performs topology analysis, file classification, semantic extraction, and dependency mapping in a single persistent session. Produces a unified codebase context document. See [BEH-SF-023](./behaviors/BEH-SF-017-agent-roles.md).

**Coverage Agent** (`coverage-agent`)
Consolidated coverage agent that tracks spec-to-implementation coverage metrics and validates that coverage meets configured thresholds, identifying specific gaps. See [BEH-SF-024](./behaviors/BEH-SF-017-agent-roles.md).

## Execution Concepts

**Progressive Execution**
The default flow execution mode. The flow pauses after every phase, allowing the user to review phase outputs before proceeding. Within a phase, all iterations run without pausing. See also: Batch Mode.

**Batch Mode**
An opt-in flow execution mode where all phases run without pausing between them. Enabled via `--batch` CLI flag or `executionMode: "batch"` in the flow definition. Useful for CI/CD automation.

**Document Layer** _(Superseded — see ACP Message Part)_
Formerly the blackboard layer storing versioned specification documents. Now replaced by `ACPMessagePart` artifacts with `contentType: "text/markdown"` in ACP sessions. Documents are represented as ACP message parts with type `document` metadata. See [types/acp.md](./types/acp.md) and [BEH-SF-229](./behaviors/BEH-SF-229-acp-messaging.md).

**ACPRunHandle**
A reference to a running ACP agent run. Provides `sendInput(messages)`, `cancel()`, and `waitForCompletion()` methods. Returned by `ACPAgentService.createRun()`. Replaces the former `AgentHandle`.

**Claude Agent SDK**
The official programmatic SDK for Claude Code, published as `@anthropic-ai/claude-agent-sdk` (TypeScript) and `claude-agent-sdk` (Python). SpecForge uses the CLI interface (`claude -p`) rather than the SDK library directly. See [references/claude-code/agent-sdk.md](./references/claude-code/agent-sdk.md).

**ClaudeCodeBackend**
The default implementation of `AgentBackendService` that wraps the Claude Code CLI (`claude -p`) behind the ACP protocol layer. Translates ACP runs into subprocess invocations with `--output-format stream-json`, `--allowedTools`, `--mcp-config`, and `--resume` flags. Refactored from the former `ClaudeCodeAdapter`. See [behaviors/BEH-SF-239-agent-backend.md](./behaviors/BEH-SF-239-agent-backend.md) (BEH-SF-239–248).

**Message Exchange**
The service that replaces the blackboard as the communication layer between agents. Provides `postMessage()`, `getHistory()`, `getArtifacts()`, and `subscribe()` methods over ACP session history. All inter-agent communication, document artifacts, and findings flow through message exchange. See [types/ports.md](./types/ports.md).

**Tool Definition**
A declaration of a tool available to an agent, specifying name, description, parameters, and return type. Tool definitions are resolved per agent role by `ToolRegistryPort.getToolsForRole()`.

**Tool Registry**
Service managing tool discovery, registration, and role-based filtering for agent sessions. Implemented via `ToolRegistryPort`. See [types/ports.md](./types/ports.md).

## Analytical Concepts

**Delta Review**
An optimization where an agent receives only the changes since its last read, not the full ACP session history. Reduces token consumption in later iterations when documents are large and changes are incremental. See [BEH-SF-234](./behaviors/BEH-SF-229-acp-messaging.md).

**Impact Analysis**
A transitive closure query from a changed node to all affected nodes via graph relationships. Answers "what breaks if this requirement changes?" Used by the Graph Explorer (web dashboard and VS Code extension) and CLI `specforge query impact`. See [BEH-SF-006](./behaviors/BEH-SF-001-graph-operations.md).

**Orphan Node**
A graph node with no incoming or outgoing relationships. Indicates unused or disconnected artifacts — e.g., a requirement that no task traces to, or a source file that no package contains. Detected by `GraphQueryPort.orphans()`.

**Traceability Gap**
A missing link in the requirement -> task -> code -> test chain. For example, a requirement with no `TRACES_TO` task, or a task with no `TRACES_TO` source file. Detected by `GraphQueryPort.traceabilityGaps()`. See [BEH-SF-007](./behaviors/BEH-SF-001-graph-operations.md).

**Concurrency Conflict**
A collision when multiple agents or users modify the same graph node simultaneously. Resolved via last-write-wins with notification, or manual resolution. See [UX-SF-072](./capabilities/UX-SF-072-resolve-graph-concurrency-conflicts.md).

**Cold Start**
An agent session that begins without composed context from prior sessions. The default behavior when no `compositionStrategy` is configured.

**Graph Mutation Pipeline**
A configurable chain of pre-commit and post-commit hooks intercepting graph writes for validation, enrichment, and side effects. See [UX-SF-071](./capabilities/UX-SF-071-configure-graph-mutation-pipeline.md).

**Flow Engine**
The runtime component that executes flow definitions: validates definitions, creates flow-scoped ACP sessions, schedules phases, evaluates convergence, and materializes session chunks on completion. See [architecture/c3-server.md](./architecture/c3-server.md) and [behaviors/BEH-SF-057-flow-execution.md](./behaviors/BEH-SF-057-flow-execution.md).

**Reactive Graph Query**
A subscription-based query that re-evaluates and pushes updates when underlying data changes. Delivered via WebSocket. See [UX-SF-070](./capabilities/UX-SF-070-subscribe-reactive-graph-queries.md).

**Iteration**
A single pass through a phase's stages. Each iteration ends with a convergence check. A phase runs multiple iterations until converged or until `maxIterations` is reached.

**Scheduler**
The orchestration component that manages agent execution order within a phase: builds the stage DAG, dispatches agents, handles clarification routing, and collects metrics. See [BEH-SF-062](./behaviors/BEH-SF-057-flow-execution.md).

## Internal Components

**Clarification Router** (`clarification-router`)
An orchestrator component (not an agent role) that detects ACP await requests from agents needing clarification and mediates the response. Part of the scheduler's routing logic. See [BEH-SF-232](./behaviors/BEH-SF-229-acp-messaging.md).

**NLQ Translator** (`nlq-translator`)
An engine component (not an agent role) that translates natural language queries into Cypher graph queries for `specforge ask`. Handles query parsing, Cypher generation, result formatting, and follow-up context. Part of the SpecForge Server's NLQ subsystem.

## User Interfaces

**Desktop App**
Native cross-platform GUI built with Tauri (Rust + system webview). Manages the SpecForge Server as an external OS process via the Server Lifecycle Manager and renders the shared React SPA in a native window. Provides system tray integration, native OS notifications, file system watching, and auto-updates. The primary local GUI for solo mode — users install a single application and get the full stack. See [behaviors/BEH-SF-273-desktop-app.md](./behaviors/BEH-SF-273-desktop-app.md) and [architecture/c3-desktop-app.md](./architecture/c3-desktop-app.md).

**Server Discovery**
The mechanism by which clients locate a running SpecForge Server. Precedence: (1) `SPECFORGE_SERVER_URL` environment variable, (2) `.specforge/server.lock` file in the project root, (3) default `http://localhost:7654`. All clients (Desktop App, CLI, VS Code Extension, Web Dashboard) use this same discovery mechanism. See [ADR-017](./decisions/ADR-017-standalone-server-over-sidecar.md).

**Server Lifecycle Manager**
A Rust component in the Desktop App that manages the SpecForge Server as an external OS process. Starts the bundled server binary on Desktop App launch if no server is already running, performs periodic health checks, restarts the server on crash, and stops it on quit (configurable). Reads and writes the `.specforge/server.lock` file for server discovery. See [BEH-SF-281](./behaviors/BEH-SF-273-desktop-app.md) and [architecture/c3-desktop-app.md](./architecture/c3-desktop-app.md).

**Server Lock File**
A file at `.specforge/server.lock` written by the SpecForge Server on startup. Contains `pid`, `port`, and `startedAt` fields. Used by the server discovery mechanism to locate a running server instance. Deleted when the server shuts down cleanly. See [ADR-017](./decisions/ADR-017-standalone-server-over-sidecar.md).

**Tauri**
The Rust-based framework used for the Desktop App. Provides native window management, system tray, IPC bridge, file system APIs, and auto-update. Uses the operating system's webview (WebKit on macOS, WebView2 on Windows, WebKitGTK on Linux) to render the React SPA. Selected over Electron per [ADR-002](./decisions/ADR-002-tauri-over-electron.md).

**Web Dashboard**
React SPA served at localhost by the SpecForge Server. Provides the Flow Monitor (real-time flow observation), Graph Explorer (interactive knowledge graph visualization), Findings (searchable finding list), and Cost Tracker (token usage and cost analytics) views. Same codebase rendered in the Desktop App's webview and in standalone browser mode. See [behaviors/BEH-SF-133-web-dashboard.md](./behaviors/BEH-SF-133-web-dashboard.md).

**VS Code Extension**
Lightweight VS Code sidebar panels for flow status, findings, and graph queries. Communicates with the SpecForge Server via its HTTP/WebSocket API. Provides inline access to flow monitoring and graph exploration without leaving the editor. See [behaviors/BEH-SF-139-vscode-extension.md](./behaviors/BEH-SF-139-vscode-extension.md).

## Deployment & SaaS Concepts

**API Token**
A long-lived, revocable token for authenticating CLI commands and CI/CD pipelines without an interactive OAuth flow. Created in the user settings panel and stored securely. Used via `SPECFORGE_TOKEN` environment variable or `specforge login --token`. Available in SaaS mode.

**Cloud API**
The HTTPS/WebSocket API provided by SpecForge Cloud for managed infrastructure services: Neo4j proxy, OAuth identity, billing, and agent marketplace. Used only in SaaS mode. The Cloud API does not contain application logic — all flows, agents, composition, analytics, and NLQ run locally in the SpecForge Server.

**Organization (Org Context)**
A shared workspace with roles (`owner`, `admin`, `member`, `viewer`), projects, and a shared knowledge graph. Users switch between personal context and org contexts. All API calls, graph queries, and views are scoped to the active context. Available in SaaS mode.

**SaaS Mode**
Deployment where the local SpecForge Server connects to SpecForge Cloud for managed Neo4j, OAuth, billing, and agent marketplace. Same local server process as solo mode, with cloud-backed adapters for infrastructure services. Zero infrastructure setup — login and start.

**Solo Self-Hosted Mode**
Single-developer deployment. Local SpecForge Server + local Neo4j. All features, no authentication, no team management. The simplest deployment — install, start, and run flows.

**SpecForge Server**
The Node.js process that runs all SpecForge application logic: flow engine, agent session manager, ACP server, graph sync, composition engine, analytics, NLQ. Same binary in both solo and SaaS deployment modes. Launched by `specforge server start`. Mode-specific backend adapters are loaded based on `.specforge/config.json`.

## Infrastructure Concepts

**Budget Zone**
One of four progressive cost degradation levels: Green (>60% budget remaining, normal operation), Yellow (30-60%, non-critical roles downgraded), Orange (10-30%, all roles to Sonnet), Red (<10%, essential stages only). Transitions are monotonic within a phase. See [BEH-SF-170](./behaviors/BEH-SF-169-cost-optimization.md).

**Cost Prediction**
Formula-based cost estimation via `specforge estimate`. Computes `flow_cost = sum(phase_estimates)` using role model costs, estimated tokens, and historical data. Produces tabular output with confidence intervals. See [BEH-SF-171](./behaviors/BEH-SF-169-cost-optimization.md).

**Graph Sync Hook**
A PostToolUse hook that upserts file nodes in Neo4j on every `Write`, `Edit`, or file-creating `Bash` tool completion. Computes SHA-256 content hash, extracts requirement IDs via regex, and creates `CONTAINS` edges. See [BEH-SF-165](./behaviors/BEH-SF-161-hook-pipeline.md).

**Hook Pipeline**
The programmable event bus that intercepts every tool invocation across all agent sessions. Processes PreToolUse (synchronous, exit code semantics), PostToolUse (async FIFO), and Stop events. Provides compliance enforcement, graph sync, monitoring, and audit trail. See [ADR-011](./decisions/ADR-011-hooks-as-event-bus.md).

**Hook State**
Per-session rolling window state maintained in `.specforge/hook-state/{sessionId}/`. Used by behavioral monitors (loop detection, drift detection) that need multi-event context. Persists across tool invocations within a session. See [BEH-SF-167](./behaviors/BEH-SF-161-hook-pipeline.md).

**MCP Server Config**
Configuration for an MCP (Model Context Protocol) server that provides agents with external tool access. Includes server command, args, env, health check, and required tools. Assigned to roles via `RoleMcpMapping`. See [BEH-SF-193](./behaviors/BEH-SF-193-mcp-composition.md).

**Memory Curation**
The process of merging, ranking, and pruning memory content to stay within CLAUDE.md's effective limit (~200 lines). Rankings use recency, reference count, invariant status, and ADR acceptance status. See [BEH-SF-179](./behaviors/BEH-SF-177-memory-generation.md).

**Memory Rule**
A path-scoped rule file generated in `.claude/rules/` from behavior specifications. Agents working within the scoped file paths automatically receive relevant behavioral contracts. See [BEH-SF-178](./behaviors/BEH-SF-177-memory-generation.md).

**Notification Classification**
Categorization of system notifications by urgency (critical, warning, info) and routing channel (OS, browser, email, webhook). See [FEAT-SF-026](./features/FEAT-SF-026-notification-engine.md), [UX-SF-076](./capabilities/UX-SF-076-configure-notification-classification-routing.md).

**Model Routing**
Per-role, per-stage model selection strategy. Each agent role has a default model (opus/sonnet/haiku) based on task complexity, with convergence-responsive escalation on stalled metrics. See [ADR-014](./decisions/ADR-014-role-adaptive-model-routing.md).

**Orchestrator Event**
Typed domain event published by server components via `EventBusPort` for real-time monitoring and cross-component coordination. Defined in [types/extensibility.md](./types/extensibility.md).

**Package Layer**
One of 5 layered packages with strict downward-only dependency rules: Layer 1 `@specforge/protocol` (ACP types, FlowUpdate, Side pattern), Layer 2 `@specforge/connection` (subprocess lifecycle, ConnectionManager), Layer 3 `@specforge/session` (thread state, history, chunks, GraphSync), Layer 4 `@specforge/orchestration` (FlowEngine, phases, convergence, AgentRegistry), Layer 5 `@specforge/surface` (CLI, Desktop App, Web Dashboard, VS Code Extension). No upward imports — enforced by ESLint import restrictions. See [ADR-019](./decisions/ADR-019-zed-inspired-architecture.md), [architecture/ports-and-adapters.md](./architecture/ports-and-adapters.md#package-layering).

**Plugin Lazy Loading**
Deferred initialization of plugin resources until first use. Reduces startup time for projects with many plugins. See [UX-SF-074](./capabilities/UX-SF-074-configure-plugin-lazy-loading.md).

**Permission Simulation**
Zero-cost dry-run that walks a flow definition, resolves agent roles, applies all permission overlays, and produces a permission report without any LLM calls. Invoked via `specforge estimate --permissions`. See [BEH-SF-207](./behaviors/BEH-SF-201-permission-governance.md).

**Phase Hook**
Custom logic executed before or after a flow phase. Configured as `pre` or `post` handlers with access to the flow run context, phase metrics, and message exchange service. See [BEH-SF-091](./behaviors/BEH-SF-087-extensibility.md).

**Project Lifecycle State**
One of: `active`, `archived`, `suspended`, `migrating`. Controls permitted operations and project visibility. See [UX-SF-073](./capabilities/UX-SF-073-manage-project-lifecycle-states.md).

**Plugin MCP Pack**
A plugin-bundled set of MCP server configurations with pre-defined role bindings and credential requirements. When a plugin is enabled, its MCP servers are registered and available for role-based assignment. See [BEH-SF-199](./behaviors/BEH-SF-193-mcp-composition.md).

**Progressive Trust**
Trust escalation model where agents start at `restricted` tier and earn capabilities through demonstrated reliability. Four tiers: restricted → standard → elevated → autonomous. Promotions require clean iterations; anomalies trigger demotion. See [BEH-SF-202](./behaviors/BEH-SF-201-permission-governance.md).

**Scoped Permission Boundary**
An isolation constraint limiting a plugin's or agent's permission scope to a resource subset. Prevents cross-boundary access without explicit grants. See [UX-SF-075](./capabilities/UX-SF-075-configure-scoped-permission-boundaries.md).

**Sandboxed Execution**
Four-dimension isolation for agent sessions: filesystem (path scoping), network (domain allowlist), process (command allowlist), and resource (memory/CPU limits). Enforced by `ClaudeCodeBackend` and PreToolUse gates. See [BEH-SF-203](./behaviors/BEH-SF-201-permission-governance.md).

**Scope** (HexDi)
An isolated subset of the DI container. Each flow run gets its own scope, providing session-isolated adapter instances. Scoped adapters (e.g., per-flow-run ACP session, session manager) are created fresh for each flow run and disposed on completion.

**Session Recording**
Hook-based capture of agent session lifecycle events. `SessionStart` records role, model, tools, and permissions. `SessionEnd` records token usage, finding count, and duration. Records are immutable graph nodes. See [BEH-SF-168](./behaviors/BEH-SF-161-hook-pipeline.md).

**Token Budget**
A configurable limit on total tokens consumed by a flow run, phase, or agent session. When exceeded, the agent receives a `BudgetExceededError` signal and must wrap up. Default: no limit (unbounded). See [BEH-SF-073](./behaviors/BEH-SF-073-token-budgeting.md).

> **Token budget disambiguation (N01):** "Token Budget" in composition context = context window allocation (how many tokens fit). "Token Budget" in cost context = spend limit (maximum USD or tokens). Both use the same `TokenUsage` type but serve different purposes.

## Technical Terms

**Blackboard Event** _(Superseded -- see ACP Session events)_
Formerly an immutable record emitted on blackboard state changes. Now replaced by ACP session message events that drive graph sync and UI updates via `MessageExchangeService.subscribe()`. See [BEH-SF-237](./behaviors/BEH-SF-229-acp-messaging.md).

**Cypher**
The declarative graph query language used by Neo4j. All graph queries in SpecForge (direct queries, NLQ translations, impact analysis) are expressed in Cypher.

**Degraded Mode**
A system state where one or more non-critical port adapters have failed health checks. In degraded mode, the affected port's operations return descriptive errors, other ports continue normal operation, and the health check endpoint reports `degraded` status. See [INV-SF-19](./invariants/INV-SF-19-degraded-mode.md).

**FMEA (Failure Mode and Effects Analysis)**
A systematic risk assessment methodology that identifies potential failure modes, scores them by Severity, Occurrence, and Detection, and computes a Risk Priority Number (RPN = S x O x D). Used per ICH Q9 risk-based approach. See [risk-assessment/index.md](./risk-assessment/index.md).

**GAMP 5**
ISPE's Good Automated Manufacturing Practice Guide, 5th edition. Provides a risk-based framework for validating computerized systems in regulated industries. SpecForge is classified as GAMP 5 Category 5 (Custom Software).

**ICH Q9**
International Council for Harmonisation guideline on Quality Risk Management. Provides principles and tools for risk-based approaches in pharmaceutical quality systems, including FMEA methodology used in the SpecForge risk assessment.

**Convergence Criteria**
The set of conditions evaluated after each phase iteration to determine whether a phase should repeat or advance. Includes finding deltas, coverage changes, and trend direction. Expressed as a function `isConverged(metrics: PhaseMetrics) -> boolean`. See [behaviors/BEH-SF-057-flow-execution.md](./behaviors/BEH-SF-057-flow-execution.md).

**Phase**
A unit of work within a flow. Has a name, a set of agent roles (stages), convergence criteria, and a maximum iteration bound. Phases execute sequentially within a flow; stages within a phase may execute concurrently.

**Phase Metrics**
Quantitative measurements collected after each iteration: finding counts by severity, requirement coverage percentage, test pass/fail counts.

**Port** (HexDi)
A typed interface contract defining a service boundary. Ports declare what a component needs (outbound) or provides (inbound). Implementations are provided by adapters wired through the dependency graph.

**ResultAsync**
An asynchronous result monad from the [neverthrow](https://github.com/supermacro/neverthrow) library. Used as the return type for all port service methods. `ResultAsync<T, E>` represents an asynchronous computation that either succeeds with `T` or fails with a typed error `E`. Enables Rust-style error handling without exceptions.

**RPN (Risk Priority Number)**
The product of Severity, Occurrence, and Detection scores in FMEA risk assessment: RPN = S x O x D (maximum 1000). Determines risk classification: 1-60 Acceptable, 61-99 Conditionally Acceptable, 100+ Unacceptable. See [risk-assessment/index.md](./risk-assessment/index.md).

**Rendering Pipeline**
The process of transforming graph data into output formats (markdown spec documents, ADRs, RFCs, coverage reports). Graph is canonical; rendered files are derived artifacts.

**Session Chunk**
An immutable segment of an agent session's conversation, materialized in Neo4j. Contains metadata: topic, token count, summary, embedding vector for similarity search. Once materialized, chunks are never modified ([INV-SF-11](./invariants/INV-SF-11-session-chunk-immutability.md)).

**Session Manager**
Component managing agent session lifecycle: create, pause, resume, cancel, and snapshot operations. Implemented via `SessionManagerPort`. Maintains session-to-flow-run associations and token budget tracking. See [types/ports.md](./types/ports.md).

**Stage**
A unit within a phase that specifies which agent roles execute. Stages within a phase may run concurrently if they have no dependencies on each other.

**Sync-on-Write**
The strategy where ACP message events trigger immediate incremental graph updates. Each ACP message write produces an event; the graph sync layer processes it into Neo4j upserts. See [BEH-SF-237](./behaviors/BEH-SF-229-acp-messaging.md).

## Zed-Inspired Enhancements (ADR-020–024)

**Config Option**
A runtime-configurable parameter advertised by an agent via `ConfigOptionsUpdate` FlowUpdate. Agents publish available options (type, label, default value, constraints) and the FlowEngine can set values mid-session via `ConfigSetRequest`. See [ADR-022](./decisions/ADR-022-dynamic-agent-capabilities.md), [BEH-SF-512](./behaviors/BEH-SF-512-dynamic-capabilities.md).

**Content Gating**
The process of validating `FlowUpdate` content blocks against the current session's `SurfaceCapabilities`. Content types not supported by the surface are downgraded (e.g., markdown image → text link) or stripped with a warning. Implemented by `ContentGatingFilter`. See [ADR-022](./decisions/ADR-022-dynamic-agent-capabilities.md), [INV-SF-41](./invariants/INV-SF-41-surface-capability-gating.md), [BEH-SF-518](./behaviors/BEH-SF-512-dynamic-capabilities.md).

**Extension Method**
A custom protocol method registered by agents or third-party integrations. All extension method names start with `_` followed by a namespace and method name (e.g., `_acme.lint`, `_github.pr-review`). Extension results are delivered as `ExtFlowUpdate` (not `OrchestratorEvent`). See [ADR-020](./decisions/ADR-020-protocol-extension-observability.md), [INV-SF-38](./invariants/INV-SF-38-extension-method-isolation.md), [BEH-SF-496](./behaviors/BEH-SF-496-protocol-extensions.md).

**MCP Proxy**
A routing layer that exposes MCP (Model Context Protocol) servers to agent subprocesses via stdio. The proxy translates between the agent's stdio transport and the backend's actual transport (HTTP/SSE/stdio). Implemented by `McpProxyService`. See [ADR-023](./decisions/ADR-023-session-resilience-mcp-integration.md), [BEH-SF-520](./behaviors/BEH-SF-520-session-resilience.md).

**Permission Policy**
A declarative rule evaluated by `PermissionPolicyService` to determine whether an action is allowed. Policies have a priority, conditions (role, action, resource matching), and an effect (allow/deny). Evaluation is deny-by-default and decoupled from the ACP protocol layer. See [ADR-024](./decisions/ADR-024-permission-policy-architecture.md), [INV-SF-43](./invariants/INV-SF-43-permission-policy-determinism.md), [BEH-SF-528](./behaviors/BEH-SF-528-permission-policy.md).

**Protocol Handshake**
The version negotiation exchange during `initialize()` between ACPClient and an agent backend. Returns `ProtocolHandshake` containing negotiated protocol version, backend info, and negotiated capabilities. Rejects incompatible versions with `VersionNegotiationError`. See [ADR-020](./decisions/ADR-020-protocol-extension-observability.md), [BEH-SF-503](./behaviors/BEH-SF-496-protocol-extensions.md).

**Protocol Meta**
The optional `_meta` field on `ACPMessage` and `FlowUpdate` instances carrying observability metadata (W3C Trace Context trace ID, span ID, trace flags, baggage). Opaque to protocol components — only observability adapters read `_meta` values. See [ADR-020](./decisions/ADR-020-protocol-extension-observability.md), [INV-SF-39](./invariants/INV-SF-39-protocol-meta-pass-through.md), [BEH-SF-499](./behaviors/BEH-SF-496-protocol-extensions.md).

**Session Checkpoint**
A snapshot of session state created at phase boundaries or on demand. Includes serialized state, token usage, message count, and a SHA-256 integrity hash. Used for session resume and fork operations. See [ADR-023](./decisions/ADR-023-session-resilience-mcp-integration.md), [INV-SF-42](./invariants/INV-SF-42-session-checkpoint-integrity.md), [BEH-SF-526](./behaviors/BEH-SF-520-session-resilience.md).

**Session Fork**
An operation that branches a new session from an existing checkpoint, creating a new session ID while preserving the prior state. Enables parallel exploration of alternative approaches. See [ADR-023](./decisions/ADR-023-session-resilience-mcp-integration.md), [BEH-SF-525](./behaviors/BEH-SF-520-session-resilience.md).

**Session Resume**
An operation that restores a session from a checkpoint, allowing continuation from a previous state. The checkpoint's integrity hash is verified before restoration. See [ADR-023](./decisions/ADR-023-session-resilience-mcp-integration.md), [BEH-SF-524](./behaviors/BEH-SF-520-session-resilience.md).

**Slash Command**
An agent-advertised command (e.g., `/test`, `/deploy`) published via `CommandsUpdate` FlowUpdate. Commands have a name, description, and optional arguments. The command list can update dynamically based on session state. See [ADR-022](./decisions/ADR-022-dynamic-agent-capabilities.md), [BEH-SF-516](./behaviors/BEH-SF-512-dynamic-capabilities.md).

**StopReason**
A closed enum with 7 variants indicating why an agent run completed: `end_turn`, `max_tokens`, `stop_sequence`, `tool_use`, `cancelled`, `error`, `content_filtered`. All consumers must exhaustively match all variants. See [ADR-020](./decisions/ADR-020-protocol-extension-observability.md), [BEH-SF-501](./behaviors/BEH-SF-496-protocol-extensions.md).

**Surface Capabilities**
A declaration of what content types and interaction patterns a client surface supports (e.g., markdown, images, interactive code blocks, file trees). Exchanged at session creation and used by `ContentGatingFilter` to validate content blocks. See [ADR-022](./decisions/ADR-022-dynamic-agent-capabilities.md), [INV-SF-41](./invariants/INV-SF-41-surface-capability-gating.md), [BEH-SF-519](./behaviors/BEH-SF-512-dynamic-capabilities.md).

## Skill & Workflow Concepts

**Skill**
A reusable unit of agent capability loaded from 3 sources: built-in system skills, role-scoped bundles, and project-local definitions. Each skill has a name, description, tool requirements, and activation context. See [ADR-025](./decisions/ADR-025-skill-registry-architecture.md), [BEH-SF-558](./behaviors/BEH-SF-558-skill-registry.md).

**Skill Bundle**
A named collection of skills assigned to an agent role (e.g., `spec-author-bundle`). 5 built-in bundles ship with SpecForge. See [ADR-025](./decisions/ADR-025-skill-registry-architecture.md), [BEH-SF-559](./behaviors/BEH-SF-558-skill-registry.md).

**Skill Workflow**
A composable sequence of skill invocations with data flow between steps. Defined declaratively and executed by the flow engine. See [BEH-SF-564](./behaviors/BEH-SF-558-skill-registry.md).

**Spec Component Graph**
A traceability subgraph connecting spec artifacts (requirements, behaviors, features) to implementation artifacts (source files, tests, tasks) via typed relationships. Enables spec-to-code and code-to-spec navigation. See [UX-SF-083](./capabilities/UX-SF-083-explore-spec-component-traceability-graph.md).

## Integration Concepts

**Third-Party Integration**
A bidirectional sync connection between SpecForge and external project management tools (Jira, Linear, Confluence, Notion, GitHub). Managed via `IntegrationPort`. See [FEAT-SF-031](./features/FEAT-SF-031-third-party-integrations.md), [UX-SF-086](./capabilities/UX-SF-086-connect-third-party-integration.md).

**Integration Sync**
The process of synchronizing graph data with external systems. Supports push, pull, and bidirectional modes with conflict resolution. See [UX-SF-087](./capabilities/UX-SF-087-monitor-integration-sync-status.md).

## Intelligence & Maintenance Concepts

**Architecture Health Score**
A composite metric (0–100) derived from graph analysis: coupling, dependency cycles, test coverage, spec drift, and code quality. Updated after each flow run. See [FEAT-SF-033](./features/FEAT-SF-033-predictive-analytics.md), [UX-SF-088](./capabilities/UX-SF-088-view-architecture-health-score.md).

**Autonomous Maintenance**
An opt-in mode where SpecForge triggers corrective flows when drift exceeds thresholds. All changes require approval before merging. See [FEAT-SF-034](./features/FEAT-SF-034-autonomous-maintenance.md), [UX-SF-090](./capabilities/UX-SF-090-configure-autonomous-maintenance.md).

**Predictive Drift Alert**
A proactive notification when trend analysis detects spec-to-code divergence before it becomes blocking. Based on historical drift patterns and code velocity. See [FEAT-SF-033](./features/FEAT-SF-033-predictive-analytics.md), [UX-SF-089](./capabilities/UX-SF-089-review-predictive-drift-alerts.md).

**Tech Debt Score**
Quantified technical debt from graph analysis: orphan nodes, traceability gaps, stale specs, unresolved findings. Component of Architecture Health Score. See [FEAT-SF-033](./features/FEAT-SF-033-predictive-analytics.md).
