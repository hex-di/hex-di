# v4.0 Requirements — GraphBuilder Improvements

## Milestone Requirements

### API Cleanup

- [x] **API-01**: Remove `provideAsync()` method — `provide()` auto-detects async via type-level Promise detection
- [x] **API-02**: Remove `provideFirstError()` method — `provide()` returns all errors by default
- [x] **API-03**: Remove `provideUnchecked()` method — escape hatch not needed with improved type system
- [x] **API-04**: Remove `mergeWith()` method — `merge()` always uses max depth
- [x] **API-05**: Rename `withUnsafeDepthOverride()` to `withExtendedDepth()` — clearer naming

### Validations

- [ ] **VAL-01**: Bidirectional captive validation — catch captive violations regardless of registration order

### Runtime Features

- [x] **RUN-01**: Inspection summary mode — `inspect({ summary: true })` returns 7-field summary
- [x] **RUN-02**: Disposal lifecycle — adapter-level finalizers with reverse-order cleanup

## Requirement Details

### API-01: Remove provideAsync()

**Rationale:** Type-level async detection via `IsAsyncFactory<T>` already implemented. Separate method creates cognitive overhead.

**Acceptance criteria:**

- `provideAsync()` method removed from GraphBuilder
- `provide()` auto-detects async factories via return type inspection
- All existing tests updated to use unified `provide()`

### API-02: Remove provideFirstError()

**Rationale:** First-error mode rarely used. All-errors mode provides better DX.

**Acceptance criteria:**

- `provideFirstError()` method removed from GraphBuilder
- `provide()` always returns all validation errors
- Error accumulation verified in tests

### API-03: Remove provideUnchecked()

**Rationale:** Escape hatch adds API surface without commensurate value. Type system improvements make it unnecessary.

**Acceptance criteria:**

- `provideUnchecked()` method removed from GraphBuilder
- No alternative escape hatch provided
- Tests using `provideUnchecked()` updated or removed

### API-04: Remove mergeWith()

**Rationale:** Options parameter (depth selection) adds complexity. Always using max depth is the safe default.

**Acceptance criteria:**

- `mergeWith()` method removed from GraphBuilder
- `merge()` always uses `max(A.maxDepth, B.maxDepth)`
- Tests updated to use `merge()` directly

### API-05: Rename withUnsafeDepthOverride()

**Rationale:** "Unsafe" causes hesitation. "Extended" accurately describes the behavior (extending depth limit).

**Acceptance criteria:**

- Method renamed to `withExtendedDepth()`
- All call sites updated
- JSDoc updated with clear description

### VAL-01: Bidirectional Captive Validation

**Rationale:** Current validation only catches captive violations when dependency is registered first. Forward references can slip through.

**Acceptance criteria:**

- Captive violations detected regardless of registration order
- `PendingCaptiveConstraint` type tracks forward reference constraints
- Constraints validated when ports are provided
- Works across merge operations

**Flow:**

1. `provide(SingletonAdapter)` requiring unregistered `ScopedPort` → creates pending constraint
2. `provide(ScopedAdapter)` providing `ScopedPort` → validates against pending constraints → ERROR

### RUN-01: Inspection Summary Mode

**Rationale:** Full inspection returns 15+ fields. Quick summary sufficient for most use cases.

**Acceptance criteria:**

- `inspect({ summary: true })` returns `GraphSummary`
- Summary includes: adapterCount, asyncAdapterCount, isComplete, missingPorts, isValid, errors, provides
- Default behavior unchanged (full inspection)

### RUN-02: Disposal Lifecycle

**Rationale:** Services often hold resources (connections, handles) that need cleanup. Container should manage disposal.

**Acceptance criteria:**

- Optional `dispose` function on adapters
- Container tracks creation order
- `container.dispose()` calls disposers in reverse order (LIFO)
- `scope.dispose()` disposes scoped services only
- Async disposal supported (`Promise<void>`)
- Error aggregation (all disposers called even if some fail)
