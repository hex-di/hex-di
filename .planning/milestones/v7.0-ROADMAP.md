# Milestone v7.0: Distributed Tracing

**Status:** Active
**Phases:** 23-27
**Total Plans:** TBD

## Overview

Replace the isolated per-container tracing system with distributed tracing that supports cross-container propagation, W3C Trace Context, multiple export backends (OpenTelemetry, Jaeger, Zipkin, DataDog), and framework integrations (Hono, React). The existing tracing system is removed entirely -- no compatibility shims, clean break.

## Key Decisions

- **Centralized Tree-Walking Subscription** for multi-container instrumentation (NOT tracer propagation through container hierarchy)
- **No changes to @hex-di/runtime hook inheritance** -- instrumentation is external, applied via hooks
- **No runtime package changes for MVP** -- @hex-di/tracing is a standalone package
- **@hex-di/tracing lives in packages/** (runtime library, not tooling)
- **Backend packages live in packages/** (@hex-di/tracing-otel, @hex-di/tracing-jaeger, etc.)
- **Module-level span stack** for active span context propagation across containers
- **WeakMap<InspectorAPI, Container>** for MVP reverse lookup (runtime change deferred to v8.0)
- **W3C Trace Context manual implementation** (no external dependency for header parsing)
- **OTel is the universal standard** -- Jaeger/Zipkin via OTel exporters; DataDog via dd-trace

## Phases

### Phase 23: Core Tracing Package Foundation

**Goal:** Developers can import @hex-di/tracing and use a complete tracing API with port definitions, built-in adapters (NoOp, Memory, Console), W3C Trace Context propagation, and ID generation -- all with zero external dependencies

**Depends on:** Nothing (first phase of v7.0)

**Requirements:** CORE-01, CORE-02, CORE-03, CORE-04, CORE-05, CORE-06, CORE-07, CORE-08, CORE-09, CORE-10, ADPT-01, ADPT-02, ADPT-03, ADPT-04, CTX-01, CTX-02, CTX-03, CTX-04, CTX-05, CTX-06, CTX-07, PERF-03, PERF-04

**Success Criteria** (what must be TRUE):

1. Developer can `import { TracerPort, SpanExporterPort, SpanProcessorPort } from '@hex-di/tracing'` and register them via createPort()/createAdapter() in a container graph
2. Developer can use NoOpTracer for production with zero measurable overhead (no allocations, no timing calls, singleton frozen span)
3. Developer can use MemoryTracer in tests to collect spans and assert on them via getCollectedSpans()/clear()
4. Developer can use ConsoleTracer for development with colorized, timestamped output and minimum-duration filtering
5. Developer can extract trace context from incoming W3C traceparent/tracestate headers and inject context into outgoing headers

**Plans:** TBD

**Risk:** LOW -- port/adapter definitions follow existing HexDI patterns

### Phase 24: Container Instrumentation and Context Propagation

**Goal:** Developers can instrument a single container or an entire container tree with one function call, and resolution spans automatically form parent-child relationships across container boundaries

**Depends on:** Phase 23 (requires Tracer port and adapters)

**Requirements:** INST-01, INST-02, INST-03, INST-04, INST-05, INST-06, INST-07, INST-08, INST-09

**Success Criteria** (what must be TRUE):

1. Developer calls `instrumentContainer(container, tracer)` and every resolution creates a span with port name, lifetime, cached status, and container name attributes
2. Developer calls `instrumentContainerTree(root, tracer)` and all containers in the hierarchy are instrumented, including parent-child span relationships across container boundaries
3. Resolution spans form correct parent-child trees: resolving port A which depends on port B produces a span for A containing a child span for B
4. Developer can call the returned cleanup function to remove all instrumentation hooks without side effects
5. Developer can use `createTracingHook(tracer, options)` to manually register a resolution hook with filtering and attribute customization

**Plans:** TBD

**Risk:** MEDIUM -- span stack for cross-container context has no JS DI prior art; needs careful testing

### Phase 25: OpenTelemetry Backend and Export Pipeline

**Goal:** Developers can export HexDI trace data to any OpenTelemetry-compatible backend (Jaeger, Zipkin, DataDog, Grafana Tempo) through dedicated packages with proper batching, resource metadata, and semantic conventions

**Depends on:** Phase 23 (requires SpanExporter/SpanProcessor ports and SpanData type)

**Requirements:** OTEL-01, OTEL-02, OTEL-03, OTEL-04, OTEL-05, OTEL-06, OTEL-07, OTEL-08, BACK-01, BACK-02, BACK-03, BACK-04

**Success Criteria** (what must be TRUE):

1. Developer installs @hex-di/tracing-otel and can export HexDI spans to any OTLP endpoint with configurable batching, resource metadata (service.name, version, environment), and semantic convention mapping
2. Developer installs @hex-di/tracing-jaeger and spans appear in Jaeger UI with correct trace hierarchy and hex-di.\* attributes
3. Developer installs @hex-di/tracing-zipkin and spans appear in Zipkin UI with correct service metadata
4. Developer installs @hex-di/tracing-datadog and spans appear in DataDog APM via dd-trace integration
5. BatchSpanProcessor shuts down cleanly with timeout (no disposal deadlock) and SimpleSpanProcessor works synchronously for testing

**Plans:** TBD

**Risk:** MEDIUM -- OTel version pinning, type bridging without casts

### Phase 26: Breaking Change Migration

**Goal:** All references to the old tracing system (TraceCollector, TracingAPI, ResolutionSpan, MemoryCollector, NoOpCollector) are removed from every package, and the full monorepo builds, typechecks, and tests clean

**Depends on:** Phases 23-25 (new tracing system must be fully available before removing old one)

**Requirements:** MIGR-01, MIGR-02, MIGR-03, MIGR-04, MIGR-05, MIGR-06, MIGR-07, MIGR-08, MIGR-09

**Success Criteria** (what must be TRUE):

1. No references to TraceCollector, TracingAPI, ResolutionSpan, MemoryCollector, NoOpCollector, or CompositeCollector exist anywhere in the codebase
2. `container.tracer`, `trace()`, and `enableTracing()` are removed from @hex-di/runtime public API
3. Integration packages (@hex-di/hono, @hex-di/react) use the new @hex-di/tracing system
4. All examples demonstrate the new tracing API (no old patterns)
5. `pnpm -r typecheck && pnpm -r test` passes with zero failures related to tracing changes

**Plans:** TBD

**Risk:** HIGH -- breaking changes across multiple packages, must coordinate carefully

---

### Phase 27: Framework Integrations and Testing Utilities

**Goal:** Developers using Hono or React get first-class tracing integration out of the box, test authors get assertion helpers for span verification, and performance benchmarks confirm acceptable overhead

**Depends on:** Phase 24 (requires instrumentation API), Phase 26 (requires clean codebase without old tracing)

**Requirements:** FRMW-01, FRMW-02, FRMW-03, FRMW-04, FRMW-05, FRMW-06, TEST-01, TEST-02, TEST-03, TEST-04, PERF-01, PERF-02

**Success Criteria** (what must be TRUE):

1. Hono developer adds `tracingMiddleware()` and incoming requests automatically get root spans with trace context extracted from headers and injected into responses
2. React developer wraps app in `<TracingProvider>` and accesses tracer/active span via `useTracer()`, `useSpan()`, and `useTracedCallback()` hooks
3. Test author creates a MemoryTracer with `createMemoryTracer()` and verifies spans using `assertSpanExists(spans, matcher)` and pattern matchers (hasAttribute, hasEvent, hasStatus, hasDuration)
4. NoOp tracer overhead is verified at less than 5% vs no tracing, and Memory tracer overhead is verified at less than 10% vs no tracing, via benchmark suite

**Plans:** TBD

**Risk:** MEDIUM -- React concurrent rendering needs validation
