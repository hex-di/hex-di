# Roadmap: v4.0 GraphBuilder Improvements

## Overview

This milestone consolidates the GraphBuilder API by removing deprecated methods, enhances runtime capabilities with inspection summaries and disposal lifecycle, and strengthens validation with bidirectional captive detection. The focus is on API cleanup (high impact, low complexity) and targeted runtime enhancements that leverage existing infrastructure.

## Phases

- [x] **Phase 12: API Cleanup** - Remove deprecated methods and improve naming
- [x] **Phase 13: Runtime Features** - Add inspection summary mode and disposal lifecycle
- [x] **Phase 14: Bidirectional Captive Validation** - Catch captive violations regardless of registration order

## Phase Details

### Phase 12: API Cleanup

**Goal**: GraphBuilder API surface is minimal and clear with no deprecated methods

**Depends on**: Phase 11 (api-removal)

**Requirements**: API-01, API-02, API-03, API-04, API-05

**Success Criteria** (what must be TRUE):

1. Developer uses single `provide()` method without choosing between `provideAsync()` or `provideUnchecked()`
2. Developer calls `withExtendedDepth()` with clear naming (no "unsafe" terminology)
3. Developer uses `merge()` method that always selects safe max depth automatically
4. All tests pass with updated method names and no deprecated API usage
5. TypeScript inference works correctly for async detection without manual marking

**Plans**: 3 plans

Plans:

- [x] 12-01-PLAN.md — Remove deprecated methods and rename withUnsafeDepthOverride
- [x] 12-02-PLAN.md — Update graph package tests to use new API
- [x] 12-03-PLAN.md — Update all other package tests to use new API

### Phase 13: Runtime Features

**Goal**: Container provides quick health checks and proper resource cleanup

**Depends on**: Phase 12

**Requirements**: RUN-01, RUN-02

**Success Criteria** (what must be TRUE):

1. Developer calls `inspect({ summary: true })` and receives 7-field summary for quick validation
2. Developer defines adapter with `dispose` function and container calls it during cleanup
3. Container disposes services in reverse dependency order (LIFO)
4. Async disposal functions complete sequentially before next disposer runs
5. All disposal errors are collected and reported even if some disposers fail

**Plans**: 2 plans

Plans:

- [x] 13-01-PLAN.md — Implement inspection summary mode (RUN-01)
- [x] 13-02-PLAN.md — Verify disposal lifecycle implementation (RUN-02)

### Phase 14: Bidirectional Captive Validation

**Goal**: Captive dependency violations detected regardless of registration order

**Depends on**: Phase 13

**Requirements**: VAL-01

**Success Criteria** (what must be TRUE):

1. Developer registers singleton adapter requiring unregistered scoped port, then registers scoped adapter - compile error appears
2. Developer registers adapters in any order and captive violations are always detected
3. Pending constraints are validated when ports are provided
4. Captive validation works across merge operations
5. Type error messages clearly explain the captive violation and registration order issue

**Plans**: 2 plans

Plans:

- [x] 14-01-PLAN.md — Verify gap existence and debug type system behavior
- [~] 14-02-PLAN.md — Skipped (gap does not exist - existing implementation sufficient)

**Outcome**: Gap hypothesis disproven. Existing `FindReverseCaptiveDependency` already catches forward reference captive dependencies. No implementation needed.
