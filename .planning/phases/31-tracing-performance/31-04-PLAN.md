  <action>
    Create a minimal object pool for reusable span objects:
    - Generic `ObjectPool<T>` class with configurable max size
    - Constructor takes: `create: () => T` factory, `reset: (obj: T) => void` callback, `maxSize: number`
    - `acquire()` method that returns pooled object (calling reset) or creates new one via factory
    - `release(obj: T)` method that returns object to pool if under max size
    - Max pool size limit to prevent unbounded growth (default: 1000 objects)

    Implementation should use a simple array as the pool storage. Keep it minimal — no statistics, no complex lifecycle.

  <files>packages/tracing/src/adapters/memory/tracer.ts, packages/tracing/src/adapters/memory/span.ts</files>
  <action>
    Modify Memory tracer to use object pooling for spans:
    - Create SpanPool using ObjectPool<MemorySpan> in MemoryTracer constructor
    - Modify startSpan() to acquire span from pool instead of `new MemorySpan()`
    - Add reset() method to MemorySpan that clears: _attributes (set to undefined), _events (set to undefined), _status (set to unset), _endTime (set to 0), _name (set to empty string). This is called by the pool's reset callback on release.
    - Modify span.end() to call toSpanData() first (cloning data to circular buffer), then release span back to pool
    - Keep circular buffer for SpanData storage (separate from live MemorySpan objects)

    Key insight: Pool the MemorySpan objects themselves, but still store the immutable SpanData snapshots in the circular buffer. This reduces object allocation overhead while maintaining the same external API.

  </action>
